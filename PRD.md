## 2. Executive Summary

학내 고성능 데스크톱(GPU 탑재)을 **게임 스트리밍 수준의 초저지연**으로 외부에서 사용할 수 있도록, 오픈소스 **Sunshine(Host) + Moonlight(Client)** 위에 **포털 SSO 연동 예약 시스템**과 **자동 페어링 Broker**를 결합한 통합 플랫폼을 구축한다. 기존 `remote.cnu.ac.kr`이 가진 **입력 지연**과 **선착순으로 인한 자원 고갈** 문제를 해소하여, 하드웨어·라이선스 제약 없이 학내 구성원이 통계·영상·3D 작업을 수행할 수 있는 **보편적 교육·업무권**을 보장한다.

---

## 3. 배경 및 문제 정의

학내에는 이미 고성능 데스크톱이 배치되어 있고 사용자는 `remote.cnu.ac.kr`을 통해 원격 접속할 수 있다. 그러나 다음 두 가지 한계로 인해 실제 활용도는 낮다.

- **입력 지연(Latency)**: 기존 원격 제어 방식은 네트워크 환경에 따라 입력 반응이 늦어, Adobe Suite·3D 렌더링 툴처럼 **정밀한 조작이 필요한 고사양 소프트웨어**를 다룰 때 작업 효율이 크게 저하된다.
- **예약 사용 기능 부재**: 선착순 접속 방식이라 **과제 마감·시험 기간 등 피크 시기**에 자원을 확보하지 못해 업무·학습이 중단되는 상황이 빈번하다.

결과적으로 "비싼 자원이 있어도 정작 필요할 때 못 쓰고, 써도 답답하다"는 사용자 경험이 누적되고 있다.

## 4. Goals / Non-Goals

### 4.1 Goals

- **G1.** 전용 클라이언트(Moonlight)를 통한 **게임 스트리밍 수준의 체감 성능**을 제공한다.
- **G2.** 개인 기기 사양·라이선스 비용에 관계없이 학내 구성원 누구나 고성능 컴퓨팅을 사용할 수 있는 **보편적 교육·업무권**을 보장한다.
- **G3.** 예약 기반 자원 분배로 **피크 시간대 자원 확보 가능성**을 예측 가능하게 만든다.
- **G4.** 자원 점유의 **편향(특정 PC 집중)** 을 줄이고 전체 가동률을 균등화한다.

### 4.2 Non-Goals

- 일반 사무용 가상 데스크톱(VDI) 시장 대체.
- 학외 일반 사용자 대상 SaaS 제공.
- GPU가 없는 호스트·구형 PC를 본 플랫폼에 편입.
- 모바일 전용 UX 최적화 (V1 범위 외).

## 5. 페르소나

### 페르소나 ① — 박지원 (대학원생 · 통계/연구형)

- **상황**: 통계학과 석사 2학기. SPSS·SAS로 대규모 데이터 분석. 개인 노트북은 RAM 부족.
- **목표**: 자택에서도 학내 PC 자원으로 분석 스크립트를 안정적으로 돌리고 싶다.
- **불편 사항(Pain)**: 마감 전날 `remote.cnu.ac.kr`에 접속해도 빈 자리가 없거나, 접속해도 입력이 느려 스크립트 디버깅이 어렵다.

### 페르소나 ② — 김서연 (학부생 · 영상/디자인형)

- **상황**: 디자인학과 4학년. Adobe Suite, 3D 렌더링 툴 사용. 라이선스 비용 부담으로 학내 인프라에 의존.
- **목표**: 카페·자택에서도 학내 라이선스 소프트웨어를 별도 설치 없이 즉시 쓰고 싶다.
- **불편 사항(Pain)**: 영상 편집은 입력 지연에 특히 민감해 기존 원격 제어로는 컷 편집이 거의 불가능하다.

### 페르소나 ③ — 이상호 (정보전산원 운영 관리자)

- **상황**: 학내 고성능 데스크톱 인프라를 운영. 장애 대응·자원 분배 책임자.
- **목표**: 어느 PC가 살아 있고, 어느 PC만 과부하인지 한눈에 파악하고 싶다.
- **불편 사항(Pain)**: 현재는 사용자가 민원을 넣어야 문제를 인지. 특정 PC만 닳고 일부는 놀고 있어 자산 효율이 낮다.

## 6. 핵심 시나리오

### 시나리오 A — 자택에서 마감 전날 3D 렌더링 (페르소나 ②)

1. 김서연은 자택에서 노트북으로 `portal.cnu.ac.kr`에 로그인 상태이다.
2. 본 플랫폼 페이지로 이동하면 **별도 로그인 없이** 캘린더 뷰가 뜬다.
3. 가용 PC 리스트에서 GPU 탑재 데스크톱을 골라 22:00–24:00 슬롯을 예약한다.
4. 22:00이 되어 '접속' 버튼을 누른다 → 브라우저가 `moonlight://` 핸들러를 호출 → Moonlight 앱이 자동 실행되며 IP·토큰이 전달된다.
5. **PIN 입력 없이** 백그라운드 페어링이 끝나고 곧바로 데스크톱 화면이 스트리밍된다.
6. 23:50, 화면에 **"10분 후 세션 종료"** 알림이 뜬다. 작업물을 클라우드에 저장한다.
7. 24:00 정각 세션이 자동 종료되고, 호스트 측에서는 로그아웃·임시 파일 정리 스크립트가 실행된다.

### 시나리오 B — 공강 중 즉시 사용 (페르소나 ①)

1. 박지원은 공강 시간에 통계 분석을 잠깐 돌리고 싶다.
2. 플랫폼에 접속하면 현재 **'접속 가능'** 상태로 동적 노출된 PC 리스트가 보인다.
3. 가장 가까운 30분 슬롯을 즉시 예약하고 원클릭으로 접속한다.
4. 분석 스크립트를 실행하고 결과만 다운로드한 뒤 정시 종료한다.

### 시나리오 C — 운영 관리자의 부하 모니터링 (페르소나 ③)

1. 이상호는 관리자 대시보드에서 전 호스트의 **연결 상태·부하·세션 점유 현황**을 실시간으로 확인한다.
2. 특정 PC에 예약이 몰리는 패턴이 보이면, 예약 데이터를 근거로 분배 정책을 조정한다.

## 7. 솔루션 개요

본 플랫폼은 4개의 축으로 구성된다. (모듈 상세 설계·통신 프로토콜·배포 방식은 본 문서 범위 외)

- **Sunshine Host**: 학내 고성능 데스크톱에 설치되는 호스트 컴포넌트. **GPU 하드웨어 가속 기반 초저지연 인코딩**을 수행해 실제 PC와 유사한 반응 속도를 만든다.
- **Moonlight Client**: 다양한 OS(Windows/macOS/Linux 등)에서 동작하는 전용 클라이언트. 60fps 이상 안정 스트리밍을 제공한다.
- **Broker (중계 서버)**: 예약 관리, 동적 토큰 발급, **자동 페어링(Auto-Pairing)** 처리, 세션 라이프사이클 통제, 호스트 상태 집계를 담당하는 핵심 백엔드.
- **포털 SSO 연동 웹 포털**: 충남대 포털 계정으로 별도 가입 없이 예약·접속 토큰 발급을 받는 통합 인증 진입점.

### 사용자 여정 (텍스트 플로우)

```
[포털 SSO 로그인]
    → [캘린더에서 가용 PC + 시간 슬롯 예약]
        → [예약 시간 도래 시 '접속' 버튼 클릭]
            → [브라우저가 moonlight:// 커스텀 URL 호출]
                → [로컬 Moonlight 앱 자동 실행 + IP/토큰 자동 전달]
                    → [Broker가 Sunshine API로 백그라운드 페어링 수행]
                        → [스트리밍 세션 시작 — 사용자 입력값 0개]
                            → [종료 10분 전 알림]
                                → [예약 종료 시 세션 강제 종료 + 호스트 정리]
```

> 본 섹션은 **사용자 관점의 흐름**까지만 기술한다. Broker의 내부 모듈 구성, 인증 토큰 방식, 데이터 모델 등은 별도 설계 문서에서 정의한다.

## 8. MVP Features (In-Scope)

인스트럭션의 MVP 8개 항목을 사용자 가치 단위 5개 Feature 카드로 묶었다.

### F1. 포털 SSO 연동 예약 페이지

- **사용자 가치**: 별도 회원가입·재로그인 없이 익숙한 학교 계정으로 즉시 예약한다.
- **User Story**: *학내 구성원으로서, 충남대 포털에 로그인된 상태라면 본 시스템에 추가 로그인 없이 비어있는 고성능 PC와 시간대를 선택해 예약하고 싶다.*
- **Acceptance Criteria (서술형)**
  - 포털 SSO 세션이 유효하면 본 시스템 진입 시 추가 로그인 화면이 보이지 않는다.
  - 캘린더 뷰에서 PC와 시간 슬롯 단위 예약이 가능하다.
  - 미인증 사용자는 SSO 인증 흐름으로 자연스럽게 유도된다.

### F2. 원클릭 접속 (커스텀 URL + 자동 페어링)

- **사용자 가치**: IP·PIN을 직접 입력할 필요 없이 버튼 한 번으로 스트리밍이 시작된다.
- **User Story**: *예약자로서, 가용 PC 리스트에서 '접속' 버튼만 누르면 Moonlight 앱이 자동 실행되고 연결까지 끝나길 바란다.*
- **Acceptance Criteria**
  - '접속' 클릭 시 브라우저가 `moonlight://` 커스텀 URL을 호출해 로컬 Moonlight 앱을 실행한다.
  - 호스트 IP와 인증 정보가 클라이언트에 자동 전달된다.
  - Sunshine API를 통한 **PIN 페어링이 사용자 개입 없이 백그라운드에서 완료**된다.

### F3. 동적 접속 토큰 / 시간 제한 PIN

- **사용자 가치**: 정당한 예약자만, 정해진 시간에만 호스트에 접근할 수 있다.
- **User Story**: *시스템으로서, 예약이 승인된 사용자에 한해 해당 시간 동안만 유효한 Sunshine 접속 PIN/토큰을 자동 발급하고 싶다.*
- **Acceptance Criteria**
  - 토큰은 예약 사용자 + 예약된 호스트 + 예약된 시간 윈도우에 한해 유효하다.
  - 시간 윈도우가 끝나면 토큰은 무효화된다.
  - 토큰 발급·소비는 사용자에게 노출되지 않는다(자동 처리).

### F4. 세션 라이프사이클 관리 (사용 / cleanup · 명시-암묵 종료 분류 · grace · 환경 초기화)

- **사용자 가치**: 다음 예약자의 자원이 보장되고, 이전 사용자의 흔적이 남지 않으며, 일시적 네트워크 끊김 같은 비정상 종료에 재접속 기회가 보장된다.
- **User Story**: *예약자로서 사용 종료 전 의사 확인 모달을 받고 싶고, 비정상적으로 연결이 끊긴 경우 일정 시간 안에 재접속할 기회를 갖고 싶다. 다음 사용자로서 이전 사용자의 파일·세션 흔적이 남지 않은 깨끗한 환경을 받고 싶다.*
- **핵심 개념**:
  - **사용 구간 vs cleanup 구간**: 예약 시간 마지막 10분은 자동 정리(cleanup) 시간으로, 실 사용 시간 = 예약 시간 − 10분 (예: 14:00–15:00 예약 → 14:00–14:50 사용 / 14:50–15:00 cleanup). 캘린더 예약 확인 모달이 이를 명시 안내한다.
  - **명시 종료 (explicit)**: 사용자 의사 확인을 거친 종료 — 종료 의사 모달의 "종료" 클릭으로 정의. 모달은 세 트리거 중 하나로 노출: ① 사용 중 예약 종료 10분 전(cleanup 시작 시각)에 자동 표시, ② 사용 중 `Ctrl+Alt+Shift+Q` 단축키, ③ 호스트에서 시작→전원→시스템 종료 클릭. 명시 종료는 **사용권 반납**으로 간주된다 (재이용은 새 예약 또는 빈 PC 즉시 사용).
  - **암묵 종료 (implicit)**: 사용자 의사 확인 없이 발생한 종료 — 인터넷 연결 끊김 / 호스트 비정상 전원 종료 / Sunshine 비정상 종료 / 사용자 PC 전원 종료 / Moonlight 비정상 종료 등. 모든 비-명시 disconnect는 암묵으로 분류된다 (기본값 안전).
  - **Grace period**: 암묵 종료에 한해 최대 10분의 재접속 유예 — 같은 connect token으로 같은 호스트에 재접속 가능. cleanup 구간은 침범하지 않는다 (예약 종료 시각 10분 전을 넘어선 grace는 불가).
- **Acceptance Criteria**
  - 예약 시간 마지막 10분은 자동 정리(cleanup) 구간이며, 실 사용 시간 = 예약 시간 − 10분이다. 캘린더 예약 확인 모달에 "실 사용 시간" 및 "마지막 10분은 자동 정리 시간" 안내가 표시된다.
  - 명시 종료 트리거 3종(예약 종료 10분 전 자동 모달 / `Ctrl+Alt+Shift+Q` / 호스트 OS 셧다운 클릭) 모두 동일한 종료 의사 모달이 노출되고, "종료" 클릭 시 사용권이 반납되며 cleanup이 즉시 시작된다. cleanup 완료 시 호스트는 즉시 IDLE 상태로 풀려 다른 사용자가 즉시 사용할 수 있다.
  - 모달의 안내 문구는 (a) 종료 시 사용권이 반납된다는 점, (b) 재이용은 새 예약 또는 빈 PC 즉시 사용으로 가능하다는 점, (c) 저장하지 않은 파일은 자동 삭제된다는 점을 명시한다.
  - 암묵 종료 발생 시 분 단위 올림한 시각부터 최대 10분의 grace 후 cleanup이 시작된다. grace는 cleanup 구간(예약 종료 10분 전)을 침범하지 않는다 (이미 cleanup 구간 안에서 끊겼다면 grace 없이 즉시 cleanup).
  - cleanup 완료 시 호스트에서 Sunshine 스트림 강제 종료 + Windows 사용자 로그아웃 + 임시 파일·다운로드·클립보드·브라우저 캐시 정리 + 다음 사용자 환경 프리셋 복원이 실행된다.
  - 호스트에서 "시작 → 전원 → 시스템 종료" 클릭은 활성 예약 보유 시 일시 차단되며 (사용자에게 차단 사유 표시) 예약자의 Moonlight에 명시 종료 모달이 push된다. 모달 "종료" 클릭으로 cleanup 후 시스템이 종료된다. 활성 예약이 없으면 즉시 종료된다.

### F5. 가용 PC 동적 노출 + 관리자 모니터링 대시보드

- **사용자 가치**: 사용자에게는 "지금 쓸 수 있는 것"만 보이고, 운영자에게는 "지금 무슨 일이 일어나는지"가 보인다.
- **User Story (사용자)**: *예약 페이지에서 전원이 켜져 있고 현재 비어 있는 PC 리스트만 보고 싶다.*
- **User Story (관리자)**: *관리자로서 모든 호스트의 연결 상태·부하를 실시간으로 한 화면에서 보고 싶다.*
- **Acceptance Criteria**
  - 호스트의 **전원 상태 + 현재 사용 여부**를 감지해 '접속 가능' 호스트만 사용자 리스트에 노출한다.
  - 관리자는 전체 호스트의 연결 상태·부하를 실시간 대시보드로 확인한다.

## 10. 비기능 요구사항 (NFR)

- **성능 (Performance)**
  - 입력 지연 **20ms ~ 50ms 이내**(목표). Moonlight + Sunshine GPU 인코딩 경로를 통한 게임 스트리밍 수준 체감.
  - 60fps 이상 안정 스트리밍.
- **보안 (Security)**
  - 충남대 포털 SSO 기반 인증.
  - **시간 제한 토큰 / 일회성 PIN**. 예약자·호스트·시간 윈도우 외 사용 불가.
  - 세션 격리: 종료 시 호스트 환경 초기화.
- **가용성 (Availability)**
  - 피크 시간대 **자원 확보 성공률을 기존 대비 40% 이상** 향상.
  - 가용 호스트만 동적으로 노출하여 "접속해도 못 쓰는" 케이스 제거.
- **운영성 (Operability)**
  - 관리자 대시보드를 통한 실시간 상태 가시성.
  - 예약 데이터 분석으로 PC 가동률 균등화 정책 수립 가능.
- **접근성 / 호환성 (Accessibility & Compatibility)**
  - Moonlight가 지원하는 다양한 OS에서 동등한 접속 경험 제공.
- **사용성 (Usability)**
  - 접속까지 **사용자가 입력하는 값 0개**(IP, PIN 등 모두 자동).

## 12. 가정 · 제약 · 의존성

- **A1.** 충남대 포털(`portal.cnu.ac.kr`)과의 SSO 연동이 기관 차원에서 승인 가능하다.
- **A2.** 대상 호스트 PC는 GPU를 보유하고 Sunshine 설치/실행이 가능하다.
- **A3.** 사용자는 Moonlight 클라이언트를 설치하고 OS에 `moonlight://` 커스텀 URL 핸들러 등록이 가능하다.
- **A4.** 학내 네트워크가 외부 클라이언트의 호스트 접근을 허용하는 정책을 갖는다(또는 그렇게 합의된다).
- **A5.** 대상 호스트의 전원 상태·로그인 상태를 원격에서 감지·제어할 수 있다(Wake/Sleep, 로그아웃 스크립트 등).
- **D1.** 기존 시스템(`remote.cnu.ac.kr`)과의 병행 운영/대체 정책은 운영 주체와의 협의 결과에 따른다.

## 14. 용어집 (Glossary)

- **Sunshine** — 호스트 PC에 설치되어 GPU 가속 기반으로 화면을 인코딩·스트리밍하는 오픈소스 호스트 소프트웨어.
- **Moonlight** — Sunshine 호스트에 접속하는 오픈소스 클라이언트. 다양한 OS 지원.
- **Broker (중계 서버)** — 예약·토큰 발급·자동 페어링·세션 관리·상태 집계를 담당하는 본 플랫폼의 백엔드.
- **SSO (Single Sign-On)** — 단일 인증으로 연동 서비스 전체에 로그인되는 방식. 본 플랫폼은 충남대 포털 SSO에 연동.
- **PIN** — Sunshine이 호스트-클라이언트 페어링 시 사용하는 일회성 인증 코드. 본 플랫폼에서는 사용자에게 노출되지 않고 Broker가 자동 처리.
- **자동 페어링 (Auto-Pairing)** — Sunshine API를 활용해 PIN 입력 단계를 백그라운드에서 자동 완료하는 모듈.
- **커스텀 URL 핸들러** — `moonlight://` 같은 비표준 스킴을 OS·브라우저가 특정 로컬 앱에 연결하는 메커니즘.
